Scenario Guidance Example
Pre-release/Initial Development The early phases of a component's life are characterised by an unstable API. This initial development phase should be reflected in the version using a 0 for the major version number, beginning with 0.1.0. Increase the minor version number for every pre-release version of the component. 0.1.0 → 0.2.0
Ready for stable consumption When a component is ready for initial release, update the version to 1.0.0. 0.2.0 → 1.0.0
Fixing a non-breaking* bug When a non-breaking change is made (e.g. a bug fix or documentation update), increment the patch version. 1.0.0 → 1.0.1
Adding a non-breaking* feature When a non-breaking new feature is added, reset the patch version to 0 and increment the minor version. 1.0.1 → 1.2.0
Adding a breaking* change When a non-backwards-compatible change is made to the API, or a change is made to a component’s visual appearance, reset the minor and patch versions to 0 and increment the major version. 1.2.0 → 2.0.0
Consuming a new version of XUI Consuming a new version of XUI is like any other change to your component. Update the major, minor, or patch version as appropriate. 2.0.0 → 2.0.1
or
2.0.0 → 2.1.0
or
2.0.0 → 3.0.0
